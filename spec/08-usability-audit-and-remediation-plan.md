# 08 系统可用性评估与整改方案

本文用于评估当前 `cargo-platform` 是否可用于真实订单报货业务，并给出后续整改方案。后续 agent 修改采集、面单解析、订单行整理、商品匹配、异常处理、导出、规则包相关功能时，应优先参考本文。

如果本文与旧的“字段定义工作台 / 商品匹配工作台 / 相似面单泛化 / 指纹识别”路线冲突，以本文为准。

## 1. 总体评估结论

当前系统还不能作为可交付客户版。

它可以继续作为真实数据打样环境使用，但必须继续整改。原因不是某一个页面或某一个接口有 bug，而是系统的业务主线还没有完全收口：

- 用户采集一批面单后，系统没有稳定说明“采集了多少张面单、解析出多少订单行、哪些可导出、哪些异常、异常原因是什么、下一步去哪里处理”。
- 面单解析、订单行整理、商品匹配、异常处理、导出中心之间存在数据不同步风险。
- 多商品面单、特殊单、备注、取消商品、尺码/颜色清洗仍然不稳定。
- 商品匹配存在隐藏候选或兜底匹配的风险，用户没有确认的商品也可能被匹配。
- UI 暴露了太多技术状态、内部编号、调试表和重复按钮，用户要像程序员一样理解系统。

当前应该保留的基础能力：

- 登录、会话、工作空间基础。
- 采集器连接、采集任务、原始采集记录。
- 商品、SKU、图片、档口等业务资产。
- Excel 导出和下载基础。
- Docker 部署脚本和 `cargo-platform-data`。

当前不能继续作为核心路线的内容：

- 旧字段定义工作台作为主入口。
- 商品匹配调试工作台暴露给普通用户。
- 多页面各自读取旧中间表。
- 无规则包时仍然偷偷解析或匹配。
- 未经用户确认的隐藏商品猜测、SKU 猜测、图片猜测。

## 2. 正确产品主线

系统目标是帮助用户整理供应商报货 Excel，不是构建一个让用户调试技术规则的平台。

正确主线应为：

```text
采集记录
  -> 面单解析
  -> 订单行整理
  -> 商品/SKU/图片匹配
  -> 异常处理
  -> 导出中心
```

用户主要面对的对象应是“订单行”，而不是字段块、路径、数据库 ID 或规则调试表。

订单行标准字段：

- `商品`
- `销售属性1`
- `销售属性2`
- `数量`
- `备注`
- `图片匹配文本`

导出正常表字段：

- `商品`
- `销售属性1`
- `图片`
- `销售属性2`
- `数量`
- `备注`
- `图片匹配文本`

多商品面单必须拆成多条订单行，不能把多个商品藏在一个字段里。

## 3. 当前主要问题

### 3.1 采集记录问题

现象：

- 多处仍显示“原始记录”“打印次数”等概念，用户只关心面单数量。
- 用户采集约 160 张面单，但页面可能显示 49、57、180 等不同数字，缺少业务解释。
- `第2000多单`、`第7000多单` 这类编号容易让用户以为系统乱序或混入其他任务。

整改：

- 用户主界面统一显示“面单数量”，不再显示“打印次数”。
- raw record id、数据库 id、source path 只出现在高级诊断。
- 所有页面使用同一个任务摘要接口展示：
  - `waybill_count`
  - `order_row_count`
  - `matched_count`
  - `exception_count`
  - `special_count`
  - `exportable_count`

### 3.2 面单解析问题

现象：

- 面单解析页有时显示 0，有时解析结果和订单行整理页不同。
- 多商品面单没有稳定拆成多条订单行。
- 单商品面单被误拆成多商品。
- 特殊单、微信单、手动单、非鞋类单、取消商品没有稳定状态。
- `颜色分类:`、`尺码:`、`*1`、`1件`、括号噪声没有统一清洗。

整改：

- 面单解析必须成为唯一“原始面单 -> 订单行”的出口。
- 后续模块不能重新拆面单，不能读取旧解析结果。
- 解析输出必须包括业务字段、状态、原因和源追溯。
- 无法解析的行不能显示为空，必须显示明确原因。

### 3.3 订单行整理问题

现象：

- 与面单解析页职责重复。
- 有些表格显示空字段、旧编号、内部状态，用户不知道下一步。
- 订单行整理和商品匹配页可能看到的不是同一份数据。

整改：

- 订单行整理只消费面单解析输出。
- 只做编辑、确认、排除、批量复核。
- 不做商品匹配规则维护，不重新解析原文。

### 3.4 商品匹配问题

现象：

- 商品匹配页过于像程序员调试台。
- 商品匹配不应暴露在普通用户页面。
- 用户未定义某个商品时，系统可能通过隐藏候选匹配出来。
- 冲突没有显示具体冲突规则。
- 学习记录难搜索、难筛选、难修改。

整改：

- 商品匹配作为管理员能力，放到管理员页面。
- 商品匹配只消费已确认订单行。
- 没有学习记录或规则包时，不做隐藏匹配。
- 冲突必须展示命中的规则编号、规则名、字段、关键词和目标商品。
- 学习记录必须可搜索、可筛选、可停用、可修改、可删除、可预览影响范围。

### 3.5 异常处理问题

现象：

- 异常提示太笼统。
- 用户不知道要处理哪条异常，也不知道跳转去哪里。
- 异常页和商品匹配页存在重复表格。

整改：

- 异常处理页只显示需要处理的问题。
- 每条异常必须有：
  - 异常类型
  - 订单行摘要
  - 原因
  - 建议处理
  - 跳转目标
- 异常按业务原因分组，不按技术表分组。

### 3.6 导出问题

现象：

- 用户无法确认导出是否丢行。
- 正常行、异常行、排除行之间缺少总数校验。

整改：

- 导出前必须做预检。
- 正常 sheet 行数 + 异常 sheet 行数 + 排除行数必须等于复核后的订单行总数。
- 不允许静默丢行。

### 3.7 规则包问题

现象：

- 规则包入口不够明显。
- 当前鞋类识别行为有混入系统默认逻辑的风险。
- 无规则包时，系统仍可能继续解析或匹配。

整改：

- 规则包是场景资产，必须有管理员入口。
- 没有启用规则包时，解析和匹配接口必须返回 `rule_pack_missing`。
- 当前鞋类规则必须打包成可导入、可导出、可启用、可停用的规则包。

## 4. 目标模块边界

### 4.1 采集模块

输入：采集器 payload。

输出：不可变原始采集记录。

不得做：

- 面单解析。
- 商品匹配。
- SKU 匹配。
- 导出。

### 4.2 面单解析模块

输入：

- 原始采集记录。
- 当前启用规则包。

输出：

- 订单行 `order_rows[]`。

每条订单行包含：

- `product`
- `sales_attr1`
- `sales_attr2`
- `quantity`
- `remark`
- `image_match_text`
- `parse_status`
- `parse_reason`
- `source_trace`
- `waybill_index`
- `item_index`
- `item_count`

不得做：

- 商品归类。
- SKU 匹配。
- 图片匹配。
- 导出。

### 4.3 订单行整理模块

输入：面单解析输出的订单行。

输出：用户确认、编辑或排除后的订单行。

不得做：

- 原始面单重新解析。
- 商品/SKU/图片规则维护。
- Excel 导出。

### 4.4 商品匹配模块

输入：

- 已确认订单行。
- 商品资产、SKU、图片资产。
- 用户确认的商品匹配学习记录。
- 当前启用规则包。

输出：

- `matched_product_id`
- `matched_sku_id`
- `matched_image_id`
- `match_status`
- `exception_reason`
- `matched_rule_ids`

不得做：

- 面单解析。
- 订单行字段修正。
- 导出。
- 隐藏兜底猜测。

### 4.5 导出模块

输入：

- 订单行。
- 商品匹配结果。

输出：供应商报货 Excel。

不得做：

- 面单解析。
- 商品匹配。
- 字段定义。

## 5. 目标页面结构

### 用户页面

1. `采集记录`
   - 看采集任务。
   - 看采集器状态。
   - 看面单数量。

2. `面单解析`
   - 看原始面单。
   - 看解析后的订单行。
   - 看特殊单和解析失败原因。
   - 支持重新解析当前任务。

3. `订单行整理`
   - 查看和编辑订单行。
   - 确认可用行。
   - 排除不需要报货的行。

4. `异常处理`
   - 查看需要处理的问题。
   - 按原因跳转处理。

5. `导出中心`
   - 预检行数。
   - 下载 Excel。

### 管理员页面

1. `规则包管理`
   - 导入、导出、启用、停用、验证规则包。

2. `商品资产`
   - 商品、SKU、图片、档口维护。

3. `商品匹配规则`
   - 维护商品/SKU/图片学习记录。
   - 搜索、筛选、修改、停用、删除规则。
   - 查看冲突规则。

4. `系统设置`
   - 账号、采集器、基础配置。

## 6. P0 整改任务

### P0-1 统一数据源

目标：所有页面读取同一份任务摘要、订单行和匹配结果。

任务：

- 审计以下页面使用的 API：
  - `采集记录`
  - `面单解析`
  - `订单行整理`
  - `商品匹配`
  - `异常处理`
  - `导出中心`
- 建立统一任务摘要接口。
- 删除或停用旧数据源读取。
- 所有页面显示同一批任务的同一份数量。

验收：

- 同一任务在所有页面显示的面单数、订单行数、异常数一致。
- 不再出现解析页 0、异常页有数据、商品匹配页又是另一份数据。

### P0-2 无规则包禁止识别

目标：没有规则包时不做隐藏识别。

任务：

- 解析接口检查 active rule pack。
- 商品匹配接口检查 active rule pack。
- 没有规则包时返回 `rule_pack_missing`。
- UI 显示“请导入或启用识别规则包”。

验收：

- 禁用所有规则包后，系统不生成新订单行，不生成新匹配结果。
- 页面提示清楚。

### P0-3 面单解析输出订单行

目标：解析页输出的是后续模块可直接消费的订单行。

任务：

- 面单解析模块统一输出 `order_rows[]`。
- 订单行保留 `source_trace`。
- 多商品输出多行。
- 特殊单输出特殊状态。
- 解析失败输出失败原因。

验收：

- 后续模块不再读取原始文本重新解析。
- 解析结果能直接进入订单行整理。

### P0-4 多商品和单商品判定

目标：多商品拆开，单商品不误拆。

必须覆盖真实样本：

- `1小79 备1 of2025, 白黑色, 40`
  - 商品：`1小79 备1 of2025`
  - 销售属性1：`白黑色`
  - 销售属性2：`40`
  - 数量：`1`

- `范33 带木one帆布kw, 木村-3M反光, 39`
  - 商品：`范33 带木one帆布kw`
  - 销售属性1：`木村-3M反光`
  - 销售属性2：`39`
  - 数量：`1`

- `低帮黑白 42 1件; 低帮深卡其 39 1件; 低帮浅绿 36 1件`
  - 拆成 3 条订单行。

- `XXXXXXXXXX` 取消项
  - 取消项不进入正常报货。
  - 有效商品保留。

验收：

- 多商品订单行 `item_count > 1`。
- 单商品不被拆成多行。
- 取消商品有明确排除原因。

### P0-5 字段清洗

目标：输出可用于商品匹配和导出的干净字段。

规则：

- `颜色分类:` 后内容优先进入销售属性1。
- `尺码:` 后内容优先进入销售属性2。
- `*1`、`1件`、`数量:1` 统一为数量 `1`。
- 尺码字段如果包含括号噪声，只保留尺码，例如 `44(偏大一码，建议拍小)` -> `44`。
- 备注不能被当作商品，除非规则包明确声明。

验收：

- `颜色分类:Cloudtilt聯名2代白黑;鞋码:41` 解析为销售属性1 `Cloudtilt聯名2代白黑`，销售属性2 `41`。
- `用户物流催促，请加急处理!` 不变成商品。

### P0-6 商品匹配去隐藏兜底

目标：只有用户确认的学习记录或商品资产规则能产生匹配。

任务：

- 移除未确认候选作为最终匹配的逻辑。
- 商品资产候选只能作为建议。
- 没有学习记录时显示“需要补商品匹配记录”。

验收：

- 用户没有定义 `VAP` 时，`秒21 vap2025` 不自动匹配到 `VAP`。
- 不再出现只有一条学习记录却显示多规则冲突。

### P0-7 冲突诊断

目标：冲突可定位。

任务：

- 匹配结果返回所有命中的规则 ID。
- 冲突列表显示规则名、命中字段、命中关键词和目标商品。
- “查看规则”能跳到对应学习记录。

验收：

- 冲突行至少能看到两个冲突来源。
- 无法定位规则的冲突不再以 toast 堆叠方式提示。

## 7. P1 整改任务

### P1-1 页面职责收口

- 商品匹配从普通用户页面移到管理员页面。
- 异常页不显示商品匹配调试表。
- 删除重复的“检查命中”按钮。
- 保留一个明确动作：`重新匹配本批次`。

### P1-2 UI 降噪

- 主界面用业务标签，例如 `第1批-第24单`。
- 高级诊断才显示 raw id、source path、JSON。
- 表格支持搜索、筛选、分页。
- 规则列表支持按商品、关键词、字段、状态筛选。

### P1-3 异常页重做

异常页按原因分组：

- 解析失败
- 需人工复核
- 商品未命中
- SKU 未命中
- 图片未命中
- 冲突
- 特殊单

每组显示：

- 行数。
- 示例。
- 建议操作。
- 处理入口。

### P1-4 规则包入口

管理员页面增加 `规则包管理`，显示：

- 当前启用规则包。
- 规则包版本。
- 适用场景。
- 导入。
- 导出。
- 启用。
- 停用。
- 验证。

## 8. P2 整改任务

### P2-1 独立解析服务

用户希望“解析面单功能单独拆出来，调整模块部署即可”。这是正确方向。

目标服务：`waybill-parser`。

职责：

- 输入原始面单和规则包。
- 输出订单行。
- 不连接业务数据库。
- 不做商品匹配。
- 不导出 Excel。

建议接口：

```text
GET /health
POST /parse
POST /batch-parse
POST /validate-rule-pack
```

后端通过 `WAYBILL_PARSER_URL` 调用解析服务。

验收：

- 调整解析规则只需要重启解析服务。
- 主系统不因为解析规则微调而整体重启。

### P2-2 回归样本库

建立真实样本 fixture，覆盖：

- 单商品标准样本。
- 多商品样本。
- 微信特殊单。
- 手写/备注单。
- `XXXXXXXXXX` 取消商品。
- `颜色分类:` 和 `尺码:`。
- 尺码括号噪声。
- 隐藏兜底商品匹配。
- 冲突规则诊断。

验收：

- 解析或匹配逻辑修改后，必须跑相关 fixture。
- 不再修一个样本坏另一个样本。

## 9. 验收清单

### 9.1 规则包

- 无启用规则包时，解析和匹配不运行。
- 有启用规则包时，页面显示规则包名称和版本。
- 规则包可导入、导出、启用、停用。
- 当前鞋类规则不是隐藏默认逻辑。

### 9.2 面单解析

- 采集约 160 张面单时，面单数量显示一致。
- 多商品拆多行。
- 单商品不误拆。
- 特殊单可见。
- 取消商品不进入报货。
- 数量、尺码、颜色规范化。
- 备注不误当商品。

### 9.3 订单行整理

- 订单行可搜索、筛选、编辑、确认、排除。
- 每条异常都有原因。
- 每条异常都有下一步。

### 9.4 商品匹配

- 未定义规则时不匹配。
- 用户定义规则后才匹配。
- 冲突显示冲突规则。
- 学习记录可搜索、筛选、修改、停用、删除。

### 9.5 导出

- 正常 sheet 表头为：
  - 商品
  - 销售属性1
  - 图片
  - 销售属性2
  - 数量
  - 备注
  - 图片匹配文本
- 异常 sheet 名称为 `异常面单`。
- 异常 sheet 单列为 `图片匹配文本`。
- 正常行 + 异常行 + 排除行 = 复核后订单行总数。

## 10. 建议实施顺序

1. 数据源审计和统一。
2. 无规则包禁止识别。
3. 面单解析输出订单行。
4. 多商品、特殊单、取消商品、备注、尺码清洗。
5. 商品匹配去隐藏兜底。

## 13. 禁止事项

不得作为整改方案：

- 恢复旧 `waybill-structure` 路线。
- 恢复旧 `match-rules` 路线作为核心。
- 恢复旧 `print-template` 路线作为核心。
- 恢复旧 `field-definitions` 路线作为核心。
- 用隐藏相似度或指纹自动决定字段。
- 硬编码某个商品业务的最终归类。
- 删除数据库。
- 删除 Docker volume。
- 清理 `cargo-platform-data`。

## 14. Agent 执行要求

每个 agent 开始任务前必须说明：

- 当前修改哪个模块。
- 输入是什么。
- 输出是什么。
- 不允许碰哪些模块。
- 是否受规则包控制。
- 如何验证。

每个 agent 完成任务后必须汇报：

- 改动文件。
- 测试结果。
- 是否需要部署。
- 是否影响规则包。
- 是否保留 `cargo-platform-data`。

如果页面逻辑和本文冲突，应优先修页面职责和数据主线，不要继续给旧工作台加按钮。
